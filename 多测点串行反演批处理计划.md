# 多测点串行反演批处理计划

## 背景
当前 `point11/run.sh` 只会按 `config.py` 中硬编码的单个测点运行一整套流程：`save_waveform.py`、`save_gated_data.py`、`inv_dls.py`、`plot_inv.py`。而用户现在希望增加一个外层 `a.sh`，循环调用 `run.sh`，使第 `i` 次调用自动切换到：

- `SOURCE_DATA_PATH = '/Users/xiechushu/project/EM_app/TEM_app/data/sy6/测点i'`
- `TARGET_POINT = '测点i'`

从而实现多测点串行反演。

现有代码里，`config.py` 在 import 时就确定了 `TARGET_POINT`、`SOURCE_DATA_PATH`、`RESULTS_DIR`，并且多个 Python 脚本都直接 `import config`。因此最安全的实现方式不是让 `a.sh` 去反复改写 `config.py`，而是让 `run.sh` 接收测点参数，并由 `config.py` 在运行时优先读取环境变量。由于 `run.sh` 每次都会启动新的 Python 进程，这种切点方式是安全的，也能避免不同测点之间的状态串扰。

## 推荐实现方案
1. 改造 `config.py`，让 `TARGET_POINT` 和 `SOURCE_DATA_PATH` 支持“环境变量优先，默认值回退”的读取方式：
   - 优先读取环境变量 `TARGET_POINT`；
   - 若未设置，则回退到当前默认测点；
   - `SOURCE_DATA_PATH` 默认根据 `TARGET_POINT` 自动拼成 `/Users/xiechushu/project/EM_app/TEM_app/data/sy6/{TARGET_POINT}`；
   - 仅在特殊场景下再允许外部显式覆盖 `SOURCE_DATA_PATH`。
2. 在 `config.py` 中增加基础校验：
   - `TARGET_POINT` 必须存在于 `point_params.json`；
   - `SOURCE_DATA_PATH` 对应目录必须存在；
   - 校验失败时立即报错退出，避免批量任务跑到一半才发现配置错误。
3. 改造 `run.sh`，使其接受一个测点参数，例如：
   - `./run.sh 测点11`
   脚本内部根据该参数导出：
   - `TARGET_POINT=测点11`
   - `SOURCE_DATA_PATH=/Users/xiechushu/project/EM_app/TEM_app/data/sy6/测点11`
   然后继续按原顺序执行各个 Python 脚本。
4. 更新 `run.sh` 的日志输出，让其打印当前测点名、数据目录和结果目录，不再显示固定的 “Point 11”。这样批量执行时能直接从终端日志判断是否正确切点。
5. 新增外层批处理脚本 `a.sh`，只负责串行调度：
   - 定义要跑的测点列表；
   - 逐个调用 `./run.sh "测点i"`；
   - 严格串行执行；
   - 任一点失败则立即停止。
6. 继续复用现有 `RESULTS_DIR = results/TARGET_POINT` 的目录策略，不新增结果目录层级。这样每个测点的中间结果与反演结果仍会自然落到各自目录中，不会互相覆盖。
7. 不采用“由 `a.sh` 直接改写 `config.py`”的方案，因为这会把仓库留在易错状态，也不利于失败后的恢复与复现。

## 关键修改文件
- `point11/config.py`
  - 让 `TARGET_POINT` / `SOURCE_DATA_PATH` 支持环境变量驱动。
  - 增加测点配置和数据目录存在性校验。
- `point11/run.sh`
  - 接收测点参数。
  - 导出环境变量后再执行现有单点流程。
  - 更新日志，使输出与当前测点一致。
- `point11/a.sh`
  - 新增批量串行调用脚本。
  - 负责循环测点并调用 `run.sh`。
- `point11/point_params.json`
  - 继续作为测点级参数来源。
  - 本次不强制改变其结构，但 `config.py` 会继续依赖它校验 `TARGET_POINT` 是否有效。

## 复用现有实现
- 复用 `run.sh` 现有的单点流程顺序：`save_waveform.py -> save_gated_data.py -> inv_dls.py -> plot_inv.py`。
- 复用 `config.py` 当前通过 `TARGET_POINT` 从 `point_params.json` 读取测点参数的组织方式。
- 复用 `RESULTS_DIR = results/TARGET_POINT` 的结果分目录机制，无需额外设计新的输出路径。
- 复用现有各 Python 脚本的 `import config` 方式，避免给每个脚本单独增加命令行参数解析。

## 验证方案
1. 先做单点验证：
   - 运行 `./run.sh 测点11`；
   - 检查是否生成 `results/测点11/data_conf.json`、`results/测点11/inversion_result.json`、相关绘图文件。
2. 再运行另一个测点，例如 `./run.sh 测点12`；
   - 确认输出写入 `results/测点12/`，且不会覆盖 `测点11` 的结果。
3. 检查 `run.sh` 日志：
   - 应明确打印当前 `TARGET_POINT`、`SOURCE_DATA_PATH`、`RESULTS_DIR`；
   - 确认日志与实际结果目录一致。
4. 运行 `./a.sh` 做批量验证：
   - 检查是否按顺序串行执行多个测点；
   - 任一点失败时是否立即停止。
5. 批量完成后，检查 `results/` 目录：
   - 每个测点应有独立目录；
   - 现有的 `plot_multi_point.py` 等后处理脚本仍应能按 `results/测点i/` 正常读取。
